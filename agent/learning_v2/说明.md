# learning_v2 说明

`agent.learning_v2` 是新版学习层的第一阶段实现，目前重点是“证据层”。它不替换 selfplay，也不直接修改 skill，而是读取 selfplay 已经生成的完整对局输出，生成更适合后续进化使用的结构化证据。

当前定位：

```text
selfplay 输出
  -> learning_v2 证据层
  -> evidence_inputs / key_decisions / decision_evidence / game_evidence / experience_candidates
  -> 后续 evolution / battle / leaderboard
```

也就是说，`learning_v2` 当前相当于旧 `agent.learning.review` 和 `agent.learning.game_analysis` 的新版替代物。旧 `agent.learning.evolution` 中的 skill 修改、battle 验证、leaderboard 记录能力暂时还没有接入。

## 1. 总体结构

当前文件结构：

```text
agent/learning_v2/
  __init__.py
  说明.md

  models.py
  io.py
  normalizer.py
  selector.py
  rubrics.py
  prompts.py
  judge.py
  report.py
  pipeline.py
  cli.py
```

各文件职责：

```text
models.py
  定义 learning_v2 的核心数据结构。
  包括 DecisionEvidenceInput、KeyDecision、DecisionEvidence、
  GameEvidence、ExperienceCandidate、EvidenceRunResult 等。

io.py
  负责读取 selfplay 游戏目录，以及写出 learning_v2 的证据文件。
  不做评估、不做筛选、不调用 LLM。

normalizer.py
  把 archive.json、agent_decisions.jsonl、game_events.jsonl、meta.json
  标准化成 DecisionEvidenceInput。
  重点是分离 player_view 和 god_view_after_game。

selector.py
  从全量 DecisionEvidenceInput 中筛出关键决策池。
  当前包含两类规则：
    1. 规则天然关键决策
    2. 转折点固定窗口内的决策

rubrics.py
  保存角色专属评估标准。
  用于提醒 judge 不同角色的目标不同，不能用同一套标准评价。

prompts.py
  构造批量 LLM judge prompt。
  prompt 中明确要求：
    过程判断只能基于 player_view；
    结果判断才允许使用 god_view_after_game；
    必须输出更优替代决策和角色目标评估。

judge.py
  调用 LLM 进行整局关键决策批量分析。
  解析 LLM 输出，生成 DecisionEvidence、GameEvidence、ExperienceCandidate。

report.py
  把结构化证据渲染成人类可读的 evidence_report.md。

pipeline.py
  learning_v2 当前总入口。
  串起 load -> normalize -> select -> judge -> report -> write。

cli.py
  命令行入口。
  支持对单个 selfplay game_dir 运行 learning_v2。
```

## 2. 核心数据流

推荐主入口：

```python
from agent.learning_v2.pipeline import run_evidence_pipeline

result = await run_evidence_pipeline(
    game_dir="runs/selfplay/<run_id>/games/<game_id>",
    model=model,
)
```

不调用 LLM，只生成标准化输入和关键决策池：

```python
result = await run_evidence_pipeline(
    game_dir="runs/selfplay/<run_id>/games/<game_id>",
    model=None,
    use_llm=False,
)
```

命令行用法：

```bash
uv run python -m agent.learning_v2.cli \
  runs/selfplay/<run_id>/games/<game_id> \
  --no-llm
```

如果要调用 LLM：

```bash
uv run python -m agent.learning_v2.cli \
  runs/selfplay/<run_id>/games/<game_id>
```

完整流程：

```text
load_game_bundle(game_dir)
  -> normalize_decisions(bundle)
  -> select_key_decisions(evidence_inputs, bundle)
  -> judge_game_evidence(model, bundle, evidence_inputs, key_decisions)
  -> render_evidence_report(result)
  -> write_evidence_outputs(result, report, output_dir)
```

## 3. 对 selfplay 层的接口

learning_v2 不负责跑对局，它只消费 selfplay 产物。

输入目录约定：

```text
runs/selfplay/<run_id>/games/<game_id>/
  archive.json
  agent_decisions.jsonl
  game_events.jsonl
  meta.json
```

其中最重要的是：

```text
archive.json
  更接近完整快照。
  包含 decisions、public_events、player_roles、winner、final_state。

agent_decisions.jsonl
  更接近 agent 决策日志。
  包含 private_reasoning、public_text、confidence、alternatives、
  rejected_reasons、memory_summary、belief_snapshot、selected_skills、
  errors、policy_adjustments。

game_events.jsonl
  用于识别转折点、死亡、放逐、技能结果和事件顺序。

meta.json
  用于补充 game_id 等元信息。
```

learning_v2 当前对 selfplay 的最小要求：

```text
必须存在：
  archive.json

建议存在：
  agent_decisions.jsonl
  game_events.jsonl
  meta.json
```

如果缺少 `agent_decisions.jsonl`，learning_v2 仍可以从 `archive.json.decisions` 中构造部分 evidence input，但会缺少更完整的 agent 私有推理字段。

如果缺少 `game_events.jsonl`，learning_v2 仍可以做规则天然关键决策筛选，但转折点窗口筛选会变弱。

### 3.1 selfplay -> learning_v2 字段契约

learning_v2 会从 selfplay 中读取并整理这些字段：

```text
decision_id
decision_index
day
phase
action_type
player_id
role
candidates
observation_summary
memory_context
belief_context
prompt_messages
selected_skills
private_reasoning
alternatives
rejected_reasons
confidence
memory_summary
raw_output
selected_target
selected_choice
public_text
final_response
source
errors
policy_adjustments
player_roles
winner
final_state
```

标准化后的核心结构是：

```text
DecisionEvidenceInput:
  decision_id
  decision_index
  day
  phase
  action_type

  player_view:
    玩家当时能看到、能记住、能推理的信息。

  agent_reasoning:
    agent 当时的私有推理、替代方案、置信度、原始输出。

  decision_result:
    agent 最终做了什么、说了什么、是否 fallback 或被 policy 修正。

  god_view_after_game:
    事后身份、胜负、目标真实身份、最终状态。
```

最重要的边界：

```text
player_view 只用于评价过程是否合理。
god_view_after_game 只用于评价结果是否好坏。
不能用上帝视角反推 agent 当时应该知道什么。
```

### 3.2 learning_v2 输出位置

默认输出到单局目录下：

```text
runs/selfplay/<run_id>/games/<game_id>/learning_v2/
```

输出文件：

```text
evidence_inputs.jsonl
  全量标准化决策输入。

key_decisions.json
  关键决策池，以及每条关键决策的入选原因。

decision_evidence.jsonl
  LLM 对关键决策的结构化评估。
  --no-llm 模式下为空。

game_evidence.json
  整局胜负路径、关键转折、信息流、阵营协作。
  --no-llm 模式下只保留基础信息。

evidence_report.md
  人类可读报告。

experience_candidates.jsonl
  经验候选。
  --no-llm 模式下为空。

raw_judge_output.txt
  LLM 原始输出。
  仅在调用 LLM 且有输出时生成。

errors.json
  错误信息。
  仅在有错误时生成。
```

## 4. 证据层的核心判断

learning_v2 的证据层不是简单评分，而是把关键决策分成样本类型。

单决策主轴：

```text
result_quality
+ process_quality
-> sample_type
```

其中：

```text
result_quality
  可以使用 god_view_after_game，但只评价结果。

process_quality
  只能使用 player_view，评价当时过程是否合理。

sample_type
  决定这条决策能不能进入经验候选。
```

样本类型：

```text
strong_positive
  好结果 + 好过程，可以作为正经验候选。

lucky_positive
  好结果 + 坏过程，不能直接学习，必须标记误导风险。

reasonable_failure
  坏结果 + 好过程，可以作为“过程正确但局势受限”的正经验候选。

true_error
  坏结果 + 坏过程，可以作为反模式候选。

execution_issue
  格式、解析、fallback、policy 修正等工程问题。
  不进入策略经验，应进入工程问题池。

low_learning_value
  信息不足或影响较小，只记录，不提炼。

unclear
  证据不足，无法判断。
```

每个关键决策还要求输出：

```text
role_specific_evaluation
  当前角色目标是什么？
  这个决策是否服务角色/阵营目标？

better_alternatives
  当时还有哪些选择？
  有没有更优选择？
  为什么更优？
  风险是什么？
  如果换一个动作，局势可能怎么变化？

information_flow_effect
  接收了什么信息？
  忽略了什么信息？
  传播了什么信息？
  是否影响了别人？
  是否误读或扭曲了信息？
```

## 5. 对 evolution 层的接口

当前 learning_v2 还没有直接调用旧 `agent.learning.evolution`。

未来建议接入方式：

```text
learning_v2 evidence outputs
  -> evolution_adapter
  -> skill_change_proposals
  -> old evolution workspace/applier/battle/leaderboard
```

不建议让旧 evolution 直接读取 learning_v2 的所有内部文件，而是建议新增一个 adapter，把 learning_v2 的经验候选转换成进化层更稳定的输入。

推荐未来新增：

```text
agent/learning_v2/evolution_adapter.py
agent/learning_v2/evolution_pipeline.py
```

### 5.1 learning_v2 -> evolution 的主要输出

最重要的接口文件是：

```text
experience_candidates.jsonl
```

每条经验候选包含：

```text
candidate_id
role
faction
candidate_type
topic
sample_source
evidence_decision_ids
scenario
conditions
recommendation
anti_pattern
risk_boundaries
counter_conditions
supporting_evidence
opposing_evidence
confidence
validation_need
misleading_risk
```

这些字段服务于后续 skill 进化：

```text
role
  决定候选经验可能作用于哪个角色 skill。

candidate_type
  决定是正向模式、反模式，还是边界提醒。

sample_source
  决定样本来源。
  例如 strong_positive、reasonable_failure、true_error、lucky_positive。

evidence_decision_ids
  保证每条经验候选能追溯到具体决策。

scenario / conditions
  说明经验适用场景和条件。

recommendation
  后续可能写入 skill 的策略建议来源。

anti_pattern
  后续可能写入 skill 的避免项来源。

risk_boundaries / counter_conditions
  防止 evolution 把单局经验过拟合成绝对规则。

confidence
  表示证据强度。
  单局默认不应超过 medium，除非是明确执行错误或强规则样本。

validation_need
  标记是否需要多局验证。

misleading_risk
  标记这条候选是否容易误导 agent。
```

### 5.2 evolution adapter 的建议职责

未来 adapter 不应该修改 skill，只负责转换数据。

建议职责：

```text
读取 experience_candidates.jsonl
过滤 low confidence / high misleading risk 的候选
按 role / candidate_type / topic 聚合候选
保留 evidence_decision_ids 和风险边界
转换成 skill proposal 所需的中间结构
```

建议中间结构：

```text
SkillChangeProposal:
  proposal_id
  role
  target_skill_file
  proposal_type
  source_candidate_ids
  evidence_decision_ids
  current_problem
  proposed_guideline
  anti_pattern
  risk_boundaries
  confidence
  validation_required
```

### 5.3 evolution pipeline 的建议职责

未来完整 learning_v2 learning 层可以是：

```text
run selfplay
  -> run learning_v2 evidence pipeline
  -> convert experience_candidates to skill proposals
  -> apply proposals to temporary skill version
  -> run battle
  -> compare baseline vs candidate
  -> write leaderboard / validation record
  -> only promote validated changes
```

也就是：

```text
证据层：
  只产证据和经验候选。

提案层：
  把经验候选变成 skill 修改提案。

应用层：
  写入临时 skill 版本。

验证层：
  battle 验证是否真的提升。

晋升层：
  只有通过验证的修改才能进入正式 skill。
```

### 5.4 不建议的接入方式

不建议：

```text
learning_v2 直接修改 skill。
learning_v2 单局证据直接写入长期 skill。
旧 evolution 直接读取 decision_evidence 的所有细节并自行解释。
把 lucky_positive 当作成功经验。
把 execution_issue 当作策略经验。
丢弃 risk_boundaries 和 counter_conditions。
```

原因：

```text
单局容易过拟合。
LLM 判断可能不稳定。
狼人杀强依赖具体局面。
没有风险边界的经验容易误导 agent。
工程问题和策略问题必须分开。
```

## 8. 后续推荐路线

建议分三步接入旧进化层：

```text
第一步：
  只做 evolution_adapter。
  把 experience_candidates 转成 skill proposal 中间结构。
  不修改 skill。

第二步：
  生成 skill 修改提案。
  仍然不自动应用，只给人类或后续流程审查。

第三步：
  接旧 evolution 的 workspace/applier/battle/leaderboard。
  只对临时 skill 版本应用提案。
  battle 验证后再决定是否晋升。
```

最终目标：

```text
learning_v2 成为完整 learning 层：

selfplay
  -> evidence
  -> experience candidates
  -> skill proposals
  -> temporary skill version
  -> battle validation
  -> leaderboard
  -> promotion
```
