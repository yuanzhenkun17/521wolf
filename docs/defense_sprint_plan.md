## 答辩演示冲刺计划

编制日期：2026-06-04。目标：1~2 周内让项目在答辩现场有最佳展示效果。

---

## 核心原则

答辩演示 10~15 分钟，评委看到的是"能不能跑、跑得怎么样、看得到什么"。工程健壮性（持久化、重连、缓存）在演示场景下用不到，全部推后。所有工作按"演示冲击力"排序。

---

## 演示剧本（倒推需要什么）

现场最理想的演示流程：

1. 打开浏览器，配置一局人机对战（自己扮演预言家或女巫）
2. 游戏运行，评委看到 12 个座位的状态面板
3. 轮到人类操作时，UI 弹出操作面板，现场做决策
4. AI 玩家决策时，评委能看到 Agent 的推理过程、置信度、使用的 skill
5. 游戏结束后，打开复盘面板，展示关键转折点和决策评分
6. （可选）展示自进化结果：多局 self-play 后的 Leaderboard 和策略改进

要让这个剧本成立，需要解决三个阻塞问题：

| 阻塞项 | 现状 | 影响 |
|--------|------|------|
| HumanPlayer 无超时 | `await future` 永久阻塞 | 现场如果操作慢了或误操作，游戏卡死 |
| 零 skill 注入 | 所有角色 baseline 为空 | AI 决策质量差，推理内容空洞 |
| 引擎状态不可见 | 药水/守护/查验无展示 | 评委看不到角色内部状态 |

---

## 实施计划（按天排）

### Day 1：HumanPlayer 超时兜底（0.5 天）

这是演示安全网。改动很小，但不做的话演示有翻车风险。

改 `engine/players.py` 的 `HumanPlayer.act()`：
- 加 `asyncio.wait_for(future, timeout=300)` 包裹
- 超时后返回安全兜底响应（随机合法目标或弃票）
- 在日志中标记超时事件

同时在前端 `HumanActionPanel.tsx` 加一个简单的倒计时显示（最后 60 秒高亮警告），让评委看到"限时机制"的存在。

### Day 2~3：种子 skill 创建与注入（2 天）

当前所有 AI Agent 的 prompt 里没有任何角色策略 skill——这是演示效果最大的短板。评委看到 Agent 推理时引用了 skill 内容（"根据悍跳策略，查杀目标应有可解释逻辑"），冲击力远大于空 prompt 下的泛泛而谈。

为 7 个角色各创建 2~3 个核心 skill，通过引导脚本写入版本库：

| 角色 | skill 文件 | 核心内容 |
|------|-----------|---------|
| 狼人 | `fake_seer.md` | 悍跳预言家的时机选择、查杀目标逻辑、应对真预言家反打 |
| 狼人 | `deep_wolf.md` | 深水倒钩策略：隐狼、配合队友铺垫、适时切割 |
| 预言家 | `claim_seer.md` | 起跳时机、警徽流设计、查验链公布节奏 |
| 预言家 | `check_priority.md` | 查验目标优先级：前置位攻击者 > 站边模糊者 > 沉默者 |
| 女巫 | `save_decision.md` | 首夜救人判断、防骗药、解药优先级 |
| 女巫 | `poison_decision.md` | 毒药时机：确定狼人 > 高嫌疑 > 不用 |
| 猎人 | `shoot_decision.md` | 开枪目标选择：最大狼坑 > 发言攻击者 > 不开枪 |
| 守卫 | `guard_strategy.md` | 守护优先级：已知神职 > 上夜被刀者 > 避免连续守同一人 |
| 村民 | `speech_analysis.md` | 发言分析框架：找逻辑矛盾、站边依据 |
| 村民 | `wolf_pit.md` | 狼坑整理方法：票型分析 + 发言攻击链 |
| 白狼王 | `explode_decision.md` | 自爆时机：带走关键神职 > 打断节奏 > 不自爆 |

每个 skill 文件包含 YAML front matter（name、role、applicable_actions、requires）和策略正文。正文 200~500 字，写清楚决策逻辑和优先级。

引导脚本 `scripts/seed_skills.py` 调用 `store.save_version()` + `store.set_baseline()` 写入版本库。

### Day 4~5：role_state 引擎改造 + Observation 暴露（2 天）

目的不是做完整的架构迁移，而是让引擎能输出每个角色的结构化状态，供前端展示。

**最小改动方案**：

1. `PlayerState` 加 `role_state: dict[str, Any]`（1 个字段）
2. `BaseRoleRule` 加 `init_role_state()` 和 `get_role_state()`（2 个方法）
3. 每个角色规则类实现这两个方法（7 个小类）
4. `GameEngine.__init__()` 中初始化各玩家的 role_state
5. 将 `GameState` 上的 `witch_antidote_available`、`witch_poison_available`、`guard_last_target` 的写入同步到对应玩家的 `role_state`（双写，不改读取，保证测试不挂）
6. `Observation` 加 `role_state` 字段，`observation_for()` 填充
7. `GameEngine.snapshot()` 新增方法，返回可序列化的完整状态快照（含各玩家 role_state）

**不做的事**：
- 不迁移 `GameState` 顶层字段的读取方（保持兼容）
- 不删 `seer_checks` 和 `pending_hunter_shots`（双写即可）
- 不改测试文件（双写保证旧测试仍然通过）

### Day 6：前端引擎状态面板（1.5 天）

新建 `EngineStatePanel.tsx` 组件，在游戏视图中展示：

- **角色状态卡片**：每个存活角色一张卡片，显示 role_state 中的关键信息
  - 女巫：解药 ✓/✗、毒药 ✓/✗
  - 守卫：上夜守护目标
  - 预言家：查验历史列表
  - 猎人：是否已开枪
- **数据来源**：从 `GET /api/games/{id}` 返回的 snapshot 中读取
- **位置**：放在现有 `StatusPanel`（玩家列表）下方或右侧

这个面板在答辩现场非常有冲击力——评委能直观看到"引擎内部状态是结构化、可追踪的"。

### Day 7：跑一局 self-play 生成演示数据（0.5 天）

用种子 skill 后的 Agent 跑 3~5 局 self-play，挑出 1~2 局精彩对局作为演示备选：
- 一局有狼人悍跳的
- 一局女巫毒对了人的
- 一局有反转的

这些数据在答辩时用于复盘展示，万一现场演示翻车可以切到预录数据。

### Day 8：演示彩排 + 应急准备（0.5 天）

- 确保 `npm run build` + `uv run` 启动正常
- 准备一份 30 秒的快速启动指南（启动后端 → 启动前端 → 开始游戏）
- 准备截图/录屏作为 fallback
- 确认 self-play 数据能正常加载展示

## 可选加分项（如果有剩余时间）

| 项目 | 工时 | 演示价值 | 说明 |
|------|------|---------|------|
| 暂停/恢复游戏 | 0.5 天 | 中 | 演示时可以暂停讲解 |

## 不做的事（推后到答辩后）

以下在原计划中但推后，因为演示场景用不到：

- SSE 心跳保活（10 分钟演示不会断连）
- GameManager 状态持久化（演示时不会重启后端）
- 断线重连机制（演示在本地，网络稳定）
- Leaderboard 标识统一（不影响演示）
- Consolidation 路径统一（不影响演示）
- Composite Skill Dir 缓存（演示只跑几局，不需要优化）
- History 剪枝（不影响演示）
- 500ms 轮询优化（不影响演示）
- GameState 字段完整迁移（双写足够，不需要删旧字段）
