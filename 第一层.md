# 第一层：狼人杀规则系统说明

本文介绍 `src/werewolf` 这一层的整体框架、各文件职责，以及当前规则系统是如何运行一局狼人杀的。

第一层的定位是“硬编码规则引擎”。它不负责玩家怎么思考，也不负责 UI 如何展示；它只负责维护游戏状态、推进白天黑夜流程、校验玩家动作是否合法、结算角色技能、记录日志，并在合适的时间判断胜负。

## 整体边界

项目目前分为三层：

1. 第一层：`src/werewolf`
   规则系统。负责游戏流程、角色技能、投票、死亡、警长、胜负、日志。

2. 第二层：`playeragent`
   玩家 agent。每个玩家根据规则层发来的 `ActionRequest` 做决策，并返回 `ActionResponse`。

3. 第三层：`ui`
   前端和后端展示层。主要读取规则层产出的日志和 agent 决策日志，把对局过程展示给用户。

第一层和第二层的核心接口只有一个：规则层调用玩家对象的 `act(request)` 方法，玩家返回一个动作结果。

```python
class PlayerAgent(Protocol):
    async def act(self, request: ActionRequest) -> ActionResponse:
        ...
```

也就是说，第一层不关心玩家是真人、LLM agent、脚本 agent，还是以后接入的 UI 人类玩家。只要能响应 `ActionRequest`，就能参与游戏。

## 核心运行链路

一局游戏从 `GameEngine.run_until_finished()` 开始。

整体流程大致是：

1. 初始化玩家、身份、状态和日志。
2. 第一夜先执行夜间技能，但不立刻公布死亡。
3. 如果启用警长，第一夜结束后进入警长竞选。
4. 警长竞选结束后，公布第一夜死亡。
5. 每天循环执行：
   黑夜行动 -> 天亮死亡结算 -> 白天发言 -> 放逐投票 -> 死亡/技能/胜负结算。
6. 任意玩家出局后都可能触发胜负检测、警徽处理、遗言、猎人开枪等后续逻辑。
7. 达成胜负后结束游戏并写入日志。

第一层的核心思路是：`engine.py` 只做总调度，细节拆到不同模块。

```text
GameEngine
  -> phases/night.py       黑夜流程
  -> phases/sheriff.py     警长竞选
  -> phases/day.py         白天发言
  -> phases/exile.py       放逐投票
  -> role_rules/*.py       各角色技能
  -> rules/*.py            通用规则
  -> actions.py            向玩家请求动作
  -> logging.py            记录日志
```

## 顶层文件职责

### `engine.py`

`engine.py` 是第一层的核心入口，但它已经尽量被拆薄。

主要职责：

- 创建并持有 `GameState`。
- 保存玩家 agent 映射：`agents: dict[int, PlayerAgent]`。
- 初始化 `GameLogger`。
- 校验身份配置、玩家人数、角色规则是否完整。
- 提供常用查询方法：
  - `alive_ids()`
  - `dead_ids()`
  - `role_ids()`
  - `team_ids()`
- 为某个玩家生成观察信息：`observation_for(player_id, metadata)`。
- 通过 `_ask()` 向第二层玩家请求动作。
- 对外暴露阶段运行方法：
  - `run_night()`
  - `run_sheriff_election()`
  - `run_day_speeches()`
  - `run_exile_vote()`
- 对外暴露死亡、警徽、猎人、遗言等规则入口。
- 在 `run_until_finished()` 中串联整局游戏。

它的设计目标是“调度中心”，而不是把所有规则都写在一个文件里。具体的规则计算大多委托给 `phases/`、`role_rules/` 和 `rules/`。

### `models.py`

定义第一层最核心的数据模型和枚举。

重要枚举：

- `Team`：狼人阵营、村民阵营、神职阵营。
- `Winner`：狼人胜利或好人胜利。
- `Role`：狼人、白狼王、村民、预言家、女巫、猎人、守卫。
- `Phase`：准备、警长竞选、黑夜、白天发言、放逐投票、结束。
- `ActionType`：规则层可能要求玩家执行的动作。
- `DeathCause`：死亡原因。

重要数据类：

- `PlayerState`
  单个玩家在规则层中的状态，包括座位号、角色、是否存活。

- `DeathRecord`
  一条死亡记录，包括死亡玩家、原因、天数、阶段。

- `Observation`
  规则层给某个玩家看的信息。这里会做视角隔离，例如狼人可以看到狼队友，预言家可以看到自己的查验记录。

- `ActionRequest`
  第一层发给第二层玩家 agent 的动作请求。

- `ActionResponse`
  第二层玩家 agent 返回给第一层的动作结果。

- `GameState`
  整局游戏的当前状态，包括玩家、天数、阶段、公开日志、死亡列表、警长、女巫药品、守卫上一次守护目标、预言家查验结果、待处理遗言、待处理猎人开枪、胜利方等。

### `config.py`

定义可配置的游戏规则参数。

核心是 `GameConfig`：

- `name`：配置名称。
- `role_counts`：角色数量配置。
- `enable_sheriff`：是否启用警长。
- `night_order`：夜晚角色行动顺序。

当前默认配置是 `STANDARD_12`，也就是 12 人标准局：

- 3 狼人
- 1 白狼王
- 4 村民
- 1 预言家
- 1 女巫
- 1 猎人
- 1 守卫

后续如果要扩展 9 人局、8 人局、娱乐局，应该优先新增 `GameConfig`，而不是在 `engine.py` 里写死人数和身份。

### `roles.py`

负责身份分配。

主要函数：

- `roles_from_config(config)`
  根据 `GameConfig.role_counts` 展开角色列表。

- `assign_roles(config, seed=None, rng=None, ordered_roles=None)`
  生成座位号到角色的映射。默认随机打乱，测试时也可以传入固定身份顺序。

- `random_standard_roles(seed=None, rng=None)`
  使用 `STANDARD_12` 随机分配 12 人局身份。

- `standard_roles()`
  固定顺序身份，主要适合测试和可复现规则模拟。

### `players.py`

定义第一层眼里的“玩家接口”。

- `PlayerAgent`
  一个 `Protocol`，要求玩家对象实现 `act(request)`。

- `ScriptedAgent`
  测试用脚本玩家。它可以预设一串响应，用来稳定测试规则系统。

这里是第一层和第二层解耦的关键。规则层不依赖 LLM，也不依赖具体策略，只依赖这个协议。

### `actions.py`

负责规则层向玩家请求动作。

核心函数是 `ask()`，流程是：

1. 构造 `ActionRequest`。
2. 调用 `engine.agents[player_id].act(request)`。
3. 支持同步或异步返回。
4. 校验返回的 `ActionResponse` 是否符合当前动作要求。
5. 合法则写入内部事件、公开信息和详细日志。
6. 非法则重试。
7. 重试仍失败则使用默认动作。

这里也是规则安全边界。即使 agent 返回了奇怪结果，规则层也会用 `validator` 和 `default` 保证游戏不会崩。

例如：

- 放逐投票只能投给候选人或弃票。
- 女巫只有有药时才能救或毒。
- 守卫不能连续守同一个人。
- 白狼王只能带走非狼人目标。

### `logging.py`

负责完整对局日志。

主要对象：

- `GameLogEntry`
  单条日志，包含 index、day、phase、event_type、message、actor、target、payload 等。

- `GameLogger`
  保存日志列表，并支持导出：
  - `to_jsonl()`
  - `to_text()`
  - `write_jsonl(path)`
  - `write_text(path)`

- `next_game_log_name(log_dir)`
  生成 `game1`、`game2` 这类递增日志名。

UI 第三层主要读取 `logs/gameX.jsonl` 来展示游戏过程。

### `public_log.py`

负责写入玩家可见的公开信息。

`append_public_event()` 会把公开事件写成 JSON 字符串，放入 `engine.state.public_log`。

这份公开日志会进入 `Observation.public_log`，给第二层 agent 作为“记忆输入”的来源之一。

公开日志和完整日志不同：

- 完整日志给上帝视角、调试、UI 使用。
- 公开日志给玩家视角使用，避免泄露隐藏信息。

### `llm_agents.py`

这是兼容入口。

当前真正的第二层实现已经迁移到项目根目录的 `playeragent/`。`src/werewolf/llm_agents.py` 只是把 `playeragent` 里的 LLM agent、配置加载、模型适配器重新导出，方便旧代码继续使用。

这不是规则层核心逻辑。

### `demo_llm.py`

命令行运行一局 LLM 对局的入口。

流程：

1. 随机分配身份。
2. 创建 `AgentDecisionRecorder`。
3. 创建 12 个 LLM agent。
4. 创建 `GameEngine`。
5. 调用 `run_until_finished()`。
6. 写入：
   - `logs/gameX.jsonl`
   - `logs/gameX.txt`
   - `logs/gameX.agent.jsonl`

其中 `.agent.jsonl` 是第二层玩家决策过程日志，供上帝视角调试和 UI 展开“决策过程”使用。

### `__init__.py`

导出第一层常用对象，例如：

- `GameEngine`
- `ActionRequest`
- `ActionResponse`
- `ActionType`
- `Role`
- `PlayerAgent`
- `ScriptedAgent`
- `GameLogger`
- `standard_roles`
- `random_standard_roles`

它让外部可以用更短的路径导入规则层能力。

## `phases/`：游戏阶段流程

`phases/` 负责“大流程”，每个文件对应一个游戏阶段。

### `phases/night.py`

负责黑夜流程。

主要函数：

- `run_night(engine)`
  执行夜晚技能，并立刻公布死亡。

- `run_night_without_death_reveal(engine)`
  第一夜专用。执行夜晚技能，但死亡结果延后到警长竞选后公布。

- `resolve_night_actions(engine)`
  按 `engine.config.night_order` 依次执行角色夜间技能：
  1. 守卫
  2. 狼人
  3. 预言家
  4. 女巫

- `reveal_night_deaths(engine, result)`
  根据夜晚结算结果真正杀死玩家，并触发死亡后续逻辑。

夜晚死亡结算规则目前包括：

- 狼刀目标如果没有被守护且没有被救，会死亡。
- 如果同一晚既被守又被救，会死亡，也就是“奶穿”。
- 女巫毒人会造成额外死亡。
- 如果毒人目标和狼刀死亡目标相同，不重复死亡。

### `phases/sheriff.py`

负责警长竞选。

流程：

1. 所有存活玩家选择是否上警。
2. 上警玩家依次发表警上发言。
3. 上警玩家选择退水或留在警上。
4. 如果无人留在警上，则无人当选。
5. 未上警玩家投警长票。
6. 使用投票规则结算警长。

注意：当前实现里，初始上警玩家不能参与警长投票；即使后来退水，也不会变成警下投票玩家。

警长投票结果会写入公开日志，方便后续玩家记忆和 UI 展示。

### `phases/day.py`

负责白天发言。

流程：

1. 切换到 `DAY_SPEECH` 阶段。
2. 处理天亮后的待处理动作：
   - 遗言
   - 夜里被刀猎人的白天开枪
3. 如果已经产生胜负，则直接结束白天。
4. 决定发言顺序。
5. 玩家按顺序发言。
6. 每个玩家发言前，先询问其角色是否有白天打断技能。
   当前主要是白狼王自爆。

如果有警长，发言顺序由警长选择：

- `forward`：从警长后置位开始顺序发言，警长最后发言。
- `reverse`：从警长前置位开始逆序发言，警长最后发言。

### `phases/exile.py`

负责放逐投票和 PK。

流程：

1. 存活玩家对存活玩家投票，也可以弃票。
2. 投票结束后统一写入公开日志。
3. 如果单人最高票，立即放逐该玩家。
4. 被放逐者立即遗言。
5. 如果被放逐者是猎人，立即触发猎人开枪。
6. 如果平票，进入 PK。
7. PK 玩家发言。
8. 非 PK 玩家再次投票。
9. PK 单人最高票则放逐，否则无人出局。

放逐投票和 PK 投票都会使用通用投票规则，警长拥有 1.5 票权重。

## `role_rules/`：角色技能规则

`role_rules/` 负责把不同角色自己的技能从主流程里拆出来。

所有角色规则都遵循 `RoleRule` 协议：

- `visible_roles()`
  该角色能看见哪些人的身份。

- `seer_checks()`
  该角色能看见哪些查验结果。

- `night_action()`
  夜晚行动。

- `day_interrupt()`
  白天发言前是否可以打断流程。

### `role_rules/base.py`

定义 `RoleRule` 协议和 `BaseRoleRule` 默认实现。

默认角色没有额外信息、没有夜晚行动、不会白天打断。

### `role_rules/registry.py`

维护角色到规则对象的映射：

```python
ROLE_RULES = {
    Role.WEREWOLF: WerewolfRule(),
    Role.WHITE_WOLF_KING: WhiteWolfKingRule(),
    Role.VILLAGER: VillagerRule(),
    Role.SEER: SeerRule(),
    Role.WITCH: WitchRule(),
    Role.HUNTER: HunterRule(),
    Role.GUARD: GuardRule(),
}
```

`rule_for(role)` 是规则层获取角色规则的统一入口。

### `role_rules/werewolf.py`

狼人规则。

职责：

- 狼人可以在自己的观察信息里看到其他狼人和白狼王。
- 夜晚所有存活狼人分别选择刀人目标。
- 使用 `plurality()` 汇总狼刀目标。
- 多数票相同时，选择编号最小的目标。

白狼王继承狼人夜晚能力。

### `role_rules/white_wolf_king.py`

白狼王规则。

白狼王继承狼人规则，并额外拥有白天自爆打断能力。

白天发言轮到白狼王前，规则层会请求：

```python
ActionType.WHITE_WOLF_EXPLODE
```

如果白狼王选择合法目标：

1. 白狼王自己死亡，原因是 `SELF_EXPLODE`。
2. 目标死亡，原因是 `WHITE_WOLF`。
3. 触发死亡后续逻辑。
4. 白狼王发表遗言。
5. 目标没有遗言。
6. 当前白天终止，直接进入下一夜。

### `role_rules/seer.py`

预言家规则。

职责：

- 夜晚选择一名存活且不是自己的玩家查验。
- 查验结果是目标阵营：狼人阵营、村民阵营或神职阵营。
- 结果写入 `engine.state.seer_checks`。
- 预言家的观察信息中会包含自己的查验记录。

### `role_rules/witch.py`

女巫规则。

职责：

- 接收当前夜晚狼刀目标。
- 根据药品状态生成 metadata：
  - `attacked_player`
  - `can_save`
  - `can_poison`
- 女巫可以选择：
  - `save`
  - `poison`
  - `none`
- 解药和毒药各只能用一次。

当前规则中，女巫毒人和狼刀目标不同时，可能同夜两人出局。

### `role_rules/guard.py`

守卫规则。

职责：

- 夜晚选择一名存活玩家守护。
- 不能连续两晚守护同一个目标。
- 守护目标写入 `engine.state.guard_last_target`。

### `role_rules/hunter.py`

猎人规则。

猎人本身没有夜间行动，具体开枪逻辑放在 `rules/death.py` 中处理，因为猎人开枪是死亡触发规则，不是普通阶段动作。

当前猎人可以在以下死亡原因下开枪：

- 被狼人夜杀。
- 被白天放逐。

被女巫毒死、被白狼王带走等情况不能开枪。

### `role_rules/villager.py`

村民规则。

村民没有特殊技能，继承默认实现。

## `rules/`：通用规则

`rules/` 保存不属于单个阶段、也不属于单个角色的通用规则。

### `rules/voting.py`

投票结算。

主要函数：

- `resolve_votes(votes, sheriff_id, candidates=None, return_ties=False)`
  结算投票结果。警长票为 1.5 票。可以返回平票列表。

- `plurality(targets)`
  简单多数目标选择，用于狼人夜晚刀人。

### `rules/death.py`

死亡、遗言、猎人开枪等死亡后续逻辑。

核心规则：

- `kill_player()`
  设置玩家死亡，记录死亡原因，写公开日志和完整日志。

- `has_last_word()`
  当前只有放逐死亡和白狼王自爆本人有遗言。

- 夜晚被狼人刀死没有遗言。

- 被猎人开枪带走没有遗言。

- 被白狼王带走没有遗言。

- `resolve_exiled_player()`
  放逐玩家后，立即处理警徽、遗言和猎人开枪。

- `resolve_pending_daybreak_actions()`
  天亮后处理延后的遗言或猎人开枪。

- `resolve_hunter_death()`
  判断猎人是否能开枪，并执行开枪杀人。

这里是死亡规则的集中地，避免遗言、猎人、警徽等逻辑散落在多个阶段文件里。

### `rules/sheriff.py`

警徽死亡处理。

当警长死亡时：

1. 请求警长执行 `ActionType.SHERIFF_BADGE`。
2. 如果选择 `transfer` 且目标存活，则警徽移交。
3. 否则撕毁警徽。

警徽转移和撕毁都会写入日志。

### `rules/victory.py`

胜负判断。

当前胜负条件：

- 狼人全死：好人胜利。
- 村民全死：狼人胜利。
- 神职全死：狼人胜利。
- 狼人数大于等于好人人数：狼人胜利。

`check_winner()` 会在发现胜负后：

1. 设置 `engine.state.winner`。
2. 把阶段切到 `FINISHED`。
3. 写入 `game_end` 日志。

规则层会在关键节点检查胜负，尤其是玩家出局后，避免已经结束的局继续发言或投票。

## `agent_runtime/`：兼容层

`src/werewolf/agent_runtime/` 现在基本都是兼容转发文件。

早期 agent runtime 曾经在 `src/werewolf/agent_runtime` 下。后来第二层被独立到项目根目录 `playeragent/`，为了不破坏旧导入路径，这个目录保留了 re-export。

例如：

- `agent_runtime/runtime.py` 转发 `playeragent.runtime.AgentRuntime`
- `agent_runtime/memory.py` 转发 `playeragent.memory.AgentMemory`
- `agent_runtime/prompts.py` 转发 `playeragent.prompts`
- `agent_runtime/parsing.py` 转发 `playeragent.parsing`
- `agent_runtime/policies.py` 转发 `playeragent.policies`
- `agent_runtime/adapters.py` 转发 `playeragent.adapters`

这部分不再是第一层核心规则。后续如果确认没有旧代码依赖，可以考虑逐步删除或继续保留作为兼容入口。

## 第一层如何和第二层衔接

衔接点是 `ActionRequest` 和 `ActionResponse`。

规则层在某个阶段需要玩家行动时，会调用：

```python
await engine._ask(
    player_id,
    ActionType.EXILE_VOTE,
    candidates=alive,
    validator=lambda res: res.target in alive or res.target is None,
    default=ActionResponse(ActionType.EXILE_VOTE),
)
```

`_ask()` 会委托给 `actions.ask()`，再调用玩家 agent：

```python
response = engine.agents[player_id].act(request)
```

第二层只需要根据请求返回：

```python
ActionResponse(
    action_type=request.action_type,
    target=某个目标或 None,
    choice=某个选择或 None,
    text="公开发言或决策说明",
)
```

第一层会负责：

- 候选目标是否合法。
- 是否需要重试。
- 是否使用默认动作。
- 是否记录日志。
- 是否推进规则结算。

所以第二层可以专注于“这个玩家如何思考”，第一层专注于“这个动作是否符合规则，以及动作产生什么结果”。

## 第一层如何和第三层衔接

第三层 UI 主要读取第一层产生的结构化日志：

```text
logs/gameX.jsonl
```

每一行是一个 `GameLogEntry`，包含：

- `index`
- `day`
- `phase`
- `event_type`
- `message`
- `actor`
- `target`
- `payload`

UI 根据这些事件还原：

- 第几天/第几夜。
- 当前阶段。
- 存活玩家状态。
- 夜晚行动结果。
- 白天发言。
- 投票票型。
- 放逐结果。
- 警长结果。
- 游戏胜负。

第二层产生的玩家决策过程会写入：

```text
logs/gameX.agent.jsonl
```

UI 可以把这份 agent 决策日志挂在对应行动旁边，用折叠框展示“决策过程”。

## 当前规则实现重点

### 1. 第一夜警长流程

第一夜执行夜间技能后，不立即公布死亡。

顺序是：

```text
第一夜行动
  -> 警长竞选
  -> 公布第一夜死亡
  -> 白天发言
```

这对应 `engine.run_until_finished()` 中第一天的特殊处理。

### 2. 警长票 1.5 票

`rules/voting.py` 中，如果投票者是当前警长：

```python
weight = 1.5 if voter == sheriff_id else 1.0
```

放逐投票和警长竞选投票都复用这个投票结算逻辑。

### 3. 警徽移交或撕毁

警长死亡时由 `rules/sheriff.py` 处理：

- 可以移交给存活玩家。
- 也可以撕毁。

这使警徽规则不必写在夜晚、放逐、猎人、白狼王等多个地方。

### 4. 遗言规则

遗言集中在 `rules/death.py`。

当前口径：

- 被放逐者有遗言，并且立即发表。
- 白狼王自爆本人有遗言。
- 夜晚被狼人刀死没有遗言。
- 被女巫毒死没有遗言。
- 被猎人开枪带走没有遗言。
- 被白狼王带走没有遗言。

### 5. 猎人开枪规则

猎人开枪也是死亡触发规则，不放在 `HunterRule` 里。

当前口径：

- 猎人被放逐：遗言后可以开枪。
- 猎人夜里被狼人杀：天亮后可以开枪，但没有遗言。
- 猎人被女巫毒死：不能开枪。
- 猎人被白狼王带走：不能开枪。

### 6. 女巫、守卫和奶穿

夜晚死亡由 `phases/night.py` 统一结算。

当前逻辑：

- 守卫守中狼刀目标，可以防刀。
- 女巫救中狼刀目标，可以救人。
- 守卫和女巫同救同一个狼刀目标，会奶穿，目标死亡。
- 女巫毒药可以造成独立死亡。

### 7. 白狼王自爆

白狼王自爆属于白天打断逻辑，由 `role_rules/white_wolf_king.py` 实现。

当白狼王选择自爆并带人：

- 白狼王死亡。
- 目标死亡。
- 白狼王有遗言。
- 被带走的人没有遗言。
- 当前白天停止，直接进入下一夜。

## 如何扩展新人数配置

如果以后要增加 9 人局、8 人局，建议步骤是：

1. 在 `config.py` 新增一个 `GameConfig`。
2. 配置角色数量和夜晚行动顺序。
3. 使用 `assign_roles(config)` 生成身份。
4. 创建对应数量的 agent。
5. 用 `GameEngine(roles, agents, config=新配置)` 启动。

不要在 `engine.py` 中写死玩家数或身份分布。

## 如何扩展新角色

假设要新增“白痴”或“爱神”：

1. 在 `models.Role` 中新增角色枚举。
2. 如果有新动作，在 `models.ActionType` 中新增动作类型。
3. 在 `role_rules/` 下新增对应 `RoleRule`。
4. 在 `role_rules/registry.py` 注册新角色规则。
5. 如果新角色影响死亡、投票、胜负或警徽，优先把通用部分放入 `rules/`。
6. 如果新角色影响某个阶段流程，再修改对应 `phases/` 文件。
7. 在 `config.py` 的某个 `GameConfig` 中加入角色数量。
8. 增加测试，覆盖技能触发、边界条件和胜负影响。

扩展原则：

- 单个角色自己的技能放 `role_rules/`。
- 跨角色通用规则放 `rules/`。
- 阶段顺序和阶段内流程放 `phases/`。
- `engine.py` 尽量只做调度和统一入口。

## 当前架构优点

1. 规则层和玩家思考解耦。
   第一层只要求玩家实现 `act(request)`，不关心玩家内部是 LLM、真人还是脚本。

2. 规则流程和角色技能解耦。
   黑夜、白天、警长、放逐分别在 `phases/`，角色技能在 `role_rules/`。

3. 通用规则集中。
   死亡、胜负、投票、警徽都在 `rules/`，减少重复和逻辑散落。

4. 日志完整。
   第一层产出结构化 JSONL，第三层 UI 可以稳定读取。

5. 配置开始具备扩展性。
   玩家人数和身份数量来自 `GameConfig`，不是完全写死在引擎中。

## 当前仍可继续优化的方向

1. `engine.py` 仍然是全局入口，后续可以进一步引入更明确的 `GameLoop` 或 `PhaseRunner`，让主引擎更轻。

2. `phases/night.py` 对守卫、狼人、预言家、女巫有显式分支。以后角色更多时，可以把夜晚行动改成更通用的注册式执行。

3. `ActionType` 会随着角色增加变多。后续可以考虑让角色规则声明自己需要哪些动作。

4. `agent_runtime/` 目前只是兼容层，长期可以决定保留还是删除。

5. 公开日志、完整日志、agent 决策日志现在分属不同模块。后续如果 UI 复杂度继续提高，可以设计统一的事件索引或事件关联 ID，方便精确匹配每一步行动和每条决策原因。

总体来说，当前第一层已经从“一个大 engine 文件”拆成了比较清晰的规则系统：`engine.py` 负责调度，`phases/` 负责流程，`role_rules/` 负责角色，`rules/` 负责通用规则，`actions.py` 负责和玩家 agent 对接，`logging.py` 负责输出给 UI 和复盘。
