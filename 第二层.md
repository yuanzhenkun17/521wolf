# 第二层：玩家 Agent 系统说明

本文介绍项目第二层 `playeragent/` 的整体设计、各文件职责，以及玩家 agent 是如何接收规则层请求并做出行动决策的。

第二层的定位是“单个玩家的思考和行动系统”。它不负责游戏规则是否成立，也不负责 UI 如何展示；它只负责让每个玩家根据自己可见的信息、自己的身份、自己的记忆和策略，返回一个规则层能理解的动作。

## 第二层和其他层的关系

当前项目分三层：

1. 第一层：`src/werewolf`
   规则系统。它是裁判，负责推进游戏、校验动作、结算技能、判断胜负。

2. 第二层：`playeragent`
   玩家 agent。它是玩家大脑，负责记忆、判断、发言、投票、夜晚行动。

3. 第三层：`ui`
   UI 展示。它读取日志，把游戏过程和决策过程展示出来。

第二层只通过第一层定义的接口对接游戏：

```python
ActionRequest  ->  玩家 agent  ->  ActionResponse
```

第一层给玩家一个 `ActionRequest`，里面包含：

- 你是几号玩家。
- 你是什么身份。
- 当前第几天。
- 当前什么阶段。
- 当前要你执行什么动作。
- 你能选择哪些候选目标。
- 你能看到的公开日志。
- 你能看到的私有信息，例如狼人队友、预言家查验。
- 本次行动的额外 metadata。

第二层返回一个 `ActionResponse`，里面包含：

- `action_type`：响应的动作类型。
- `target`：选择的目标玩家，可以为空。
- `choice`：选择项，例如 `run`、`pass`、`save`、`poison`。
- `text`：公开发言或公开理由。

规则层会负责判断这个 response 是否合法。第二层只负责尽量做出合理决策。

## 当前是否使用 LangGraph

目前第二层没有使用 LangGraph。

当前是一个手写的线性 agent runtime：

```text
ActionRequest
  -> 读取记忆
  -> 更新主观信念
  -> 获取角色策略建议
  -> 构造 prompt
  -> 调用 LLM
  -> 解析 JSON
  -> 策略修正/兜底
  -> 记录决策
  -> 返回 ActionResponse
```

这条链路集中在 `AgentRuntime.act()`。

以后如果引入 LangGraph，最适合改造的是第二层，而不是第一层。可以把上述线性流程拆成多个 graph node，例如“记忆更新节点”“局势分析节点”“角色策略节点”“行动生成节点”“合法性检查节点”“反思记录节点”。但当前版本仍然是普通 Python pipeline。

## 一次 agent 行动的完整流程

当第一层需要某个玩家行动时，会调用：

```python
await agent.act(request)
```

如果这个玩家是 `LLMPlayerAgent`，调用会进入：

```text
LLMPlayerAgent.act()
  -> AgentRuntime.act()
```

`AgentRuntime.act()` 的主要流程是：

```python
memory_context = self.memory.build_context(request)
belief_context = self.belief.build_context(request, self.memory)
strategy_advice = self.strategy.advise(request, self.memory, self.belief)
messages = build_messages(...)
content = await self.model.complete(messages)
response = parse_action_response(request, content)
response = apply_response_policy(request, response)
decision = parse_decision_record(request, content, response)
enrich_decision(decision, belief_context, memory_context)
self.memory.remember_action(request, response, decision)
return response
```

如果模型调用失败、输出不是 JSON、解析失败，第二层不会让游戏崩溃，而是进入 fallback：

```text
模型失败
  -> fallback_response()
  -> fallback_decision()
  -> 记录错误和兜底原因
  -> 返回合法动作
```

这保证了第一层规则系统可以继续推进。

## 顶层文件职责

### `llm.py`

`llm.py` 是第二层对外最常用的入口。

核心对象是 `LLMPlayerAgent`。

它做三件事：

1. 保存玩家编号、角色、模型 client。
2. 创建并持有一个 `AgentRuntime`。
3. 在每次行动后，把 runtime 产生的 `DecisionRecord` 写入共享的 `AgentDecisionRecorder`。

关键属性：

- `runtime`
  真正执行思考流程的对象。

- `memory`
  当前玩家自己的记忆。

- `belief`
  当前玩家自己的主观判断。

- `persona`
  当前玩家的简单性格。

重要函数：

- `load_llm_client()`
  从环境变量或 `config/llm.local.json` 读取 LLM 配置。

- `create_llm_demo_agents(roles, client=None, decision_recorder=None)`
  根据座位身份创建一组 LLM 玩家。

这里要注意：创建 12 个玩家时，不是启动 12 个 Python 进程，而是在同一个 Python 进程里创建 12 个 `LLMPlayerAgent` 对象。

每个对象都有自己的：

- `AgentRuntime`
- `AgentMemory`
- `BeliefState`
- `persona`

但可以共享同一个模型 client。

### `runtime.py`

`runtime.py` 是第二层的核心。

核心类是 `AgentRuntime`。

它代表一个玩家的运行时大脑，持有：

- `player_id`
- `role`
- `model`
- `persona`
- `memory`
- `belief`
- `strategy`

一次行动时，它会依次执行：

1. `memory.build_context(request)`
   从公开日志和自身历史中构造记忆上下文。

2. `belief.build_context(request, memory)`
   根据记忆、私有信息、查验信息更新主观判断。

3. `strategy.advise(request, memory, belief)`
   根据角色和动作类型给出策略建议。

4. `build_messages(...)`
   把身份、局势、记忆、信念、策略建议组合成 LLM prompt。

5. `model.complete(messages)`
   调用模型。

6. `parse_action_response(...)`
   从模型输出解析出 `ActionResponse`。

7. `apply_response_policy(...)`
   对模型结果做最低限度的策略修正，避免明显非法或破坏流程。

8. `parse_decision_record(...)`
   解析模型的私有推理、备选目标、排除理由。

9. `enrich_decision(...)`
   补齐信念快照和记忆摘要。

10. `memory.remember_action(...)`
    把本次行动和决策记录进该玩家自己的记忆。

### `adapters.py`

模型适配层。

核心协议：

```python
class ModelAdapter(Protocol):
    async def complete(self, messages: list[dict[str, str]]) -> str:
        ...
```

只要一个模型客户端实现了 `complete(messages)`，就可以接入第二层。

当前实现是 `ChatCompletionClient`，它调用 OpenAI-compatible API：

- 默认 base URL：`https://router.shengsuanyun.com/api/v1`
- 默认模型：`ali/qwen3.5-flash`
- 默认 timeout：45 秒

`ChatCompletionClient` 使用标准 chat completions 格式：

```text
POST /chat/completions
{
  "model": "...",
  "messages": [...],
  "temperature": 0.4
}
```

这样以后要换 OpenAI、DeepSeek、Qwen、本地模型，只需要换 adapter 或配置。

### `memory.py`

玩家记忆系统。

核心类：

- `MemoryEvent`
- `AgentMemory`

每个玩家都有一份自己的 `AgentMemory`，不是全局共享。

`AgentMemory` 保存：

- `events`
  从公开日志中解析出的结构化事件。

- `self_history`
  自己过去做过什么动作。

- `decision_history`
  自己过去的私有决策过程。

- `suspicions`
  从发言、投票中提取出的怀疑对象。

- `claims_seen`
  听到的身份声明。

- `errors`
  模型调用或解析错误。

- `_seen_public_entries`
  用于去重，避免同一条公开日志重复进入记忆。

记忆更新入口是：

```python
observe_request(request)
```

它会读取：

```python
request.observation.public_log
```

也就是第一层提供的公开日志。

然后调用：

```python
parse_public_entry()
```

把 JSON 字符串形式的公开日志解析成 `MemoryEvent`。

比如公开日志：

```json
{
  "day": 1,
  "phase": "day_speech",
  "type": "speak",
  "actor": 5,
  "target": null,
  "content": "我是预言家，昨晚查验1号是狼人。"
}
```

会进入该玩家的长期记忆。

`AgentMemory` 还会尝试从文本里抽取：

- 谁怀疑谁。
- 谁声称了什么身份。
- 谁投给了谁。
- 谁死亡。

这些信息会用于后续 belief 更新和 prompt。

### `belief.py`

主观判断系统。

核心类：

- `PlayerBelief`
- `BeliefState`

每个玩家都有自己的 `BeliefState`。

`PlayerBelief` 记录某个目标玩家在当前玩家眼里的状态：

- `suspicion`
  怀疑度，0 到 1。

- `trust`
  信任度，0 到 1。

- `possible_roles`
  可能身份。

- `reasons`
  为什么这么判断。

`BeliefState` 会根据以下信息更新主观判断：

1. 公开记忆事件。
   例如投票、怀疑发言、身份声明。

2. 私有已知身份。
   例如狼人知道狼队友。

3. 预言家查验。
   例如预言家知道某人是狼人阵营。

4. 死亡状态。
   已经出局的人不再作为当前行动重点目标。

5. 自己身份。
   自己永远是完全已知，信任度 1，怀疑度 0。

它会给 prompt 提供：

```python
{
  "top_suspicions": [...],
  "self_belief": ...
}
```

也就是说，agent 不是纯随机投票，而是会根据公开日志和私有信息形成一个简单的局势判断。

### `strategies/`

角色策略建议层。

它不是最终决策，只是给 LLM 一份“当前身份应该怎么想”的建议。

统一入口：

```python
strategy_for(role)
```

返回对应角色的 `RoleStrategy`。

每个策略类实现：

```python
advise(request, memory, belief) -> StrategyAdvice
```

`StrategyAdvice` 包含：

- `goal`
  当前动作目标。

- `preferred_targets`
  建议优先考虑的目标。

- `avoid_targets`
  建议避免的目标。

- `public_stance`
  公开发言姿态。

- `private_notes`
  私有注意事项。

当前角色策略包括：

- `VillagerStrategy`
  村民依赖公开发言、票型和死亡信息，不假装有私有验人信息。

- `WerewolfStrategy`
  狼人隐藏队友关系，夜晚优先刀疑似神职或带队好人，白天把怀疑引向好人。

- `WhiteWolfKingStrategy`
  白狼王继承狼人策略，额外考虑自爆带走关键好人或神职。

- `SeerStrategy`
  预言家优先查验高风险玩家，白天围绕查验结果建立可信度。

- `WitchStrategy`
  女巫根据被刀对象、药品状态、局势选择救药/毒药，白天谨慎暴露身份。

- `GuardStrategy`
  守卫优先守护可信好人或关键神职，并避免连续守同一目标。

- `HunterStrategy`
  猎人死亡后根据怀疑度选择是否开枪，白天不宜过早暴露身份。

### `prompts.py`

Prompt 构造层。

主要职责：

1. 给玩家分配简单性格。
2. 构造 system prompt。
3. 构造 user prompt。
4. 给不同动作生成具体输出要求。

`default_persona(player_id, role)` 会按座位号轮换四种基础性格：

1. 发言简洁，优先保命。
2. 愿意竞选警长，喜欢带队归票。
3. 谨慎观察，投票跟随可信的人。
4. 攻击性稍强，会主动给出怀疑对象。

然后根据角色追加身份倾向。例如狼人要隐藏视角，预言家可以积极争警徽，女巫谨慎用药。

`build_system_prompt()` 会告诉模型：

- 你是几号玩家。
- 你是什么身份。
- 你只能根据可见信息行动。
- 你要区分公开发言和私有推理。
- 不要泄露不可公开解释的私有视角。
- 必须只输出 JSON。

`build_request_prompt()` 会把以下信息塞给模型：

- 当前阶段。
- 当前天数。
- 本次动作类型。
- 候选目标。
- 存活和死亡玩家。
- 当前警长。
- 公开局势摘要。
- 结构化事实记忆。
- 私有已知身份。
- 查验结果。
- metadata。
- 自己历史动作。
- 近期私有决策理由。
- 怀疑对象。
- 身份声明。
- Belief 主观判断。
- 角色策略建议。

最后要求模型输出 JSON：

```json
{
  "choice": "string|null",
  "target": "number|null",
  "text": "string",
  "reasoning": "string",
  "alternatives": [1, 2],
  "rejected_reasons": ["..."]
}
```

其中：

- `text` 是公开内容，会被规则层记录成公开发言或动作说明。
- `reasoning` 是私有推理，会进入 agent 决策日志。

### `parsing.py`

模型输出解析层。

主要函数：

- `load_json_object(content)`
  从模型输出中解析 JSON。即使模型前后夹了自然语言，也会尝试用正则提取 `{...}`。

- `parse_action_response(request, content)`
  把模型 JSON 转成第一层需要的 `ActionResponse`。

- `parse_decision_record(request, content, response)`
  把模型 JSON 转成 `DecisionRecord`，保存 reasoning、alternatives、rejected_reasons 等决策过程。

这一层只负责解析，不负责判断动作是否合法。

### `policies.py`

最低限度的策略修正和兜底层。

主要函数：

- `apply_response_policy(request, response)`
  对模型返回做轻量修正。

- `fallback_response(request)`
  当模型失败或输出非法时，给一个合法默认动作。

当前策略包括：

1. 如果模型选了不在 candidates 里的目标，直接 fallback。

2. 如果警长退水时自己是最后一个仍在警上的候选人，则强制 stay，避免集体退水导致无人竞选。

3. 不同动作有不同 fallback：
   - 发言类：先过。
   - 是否上警：默认不上警。
   - 退水：默认 stay。
   - 警徽处理：优先移交第一个候选人，否则撕毁。
   - 发言顺序：默认顺序。
   - 女巫：默认不用药。
   - 白狼王：默认不自爆。
   - 目标动作：默认选第一个候选目标。

注意：这里不是完整规则校验。完整规则校验仍然由第一层 `actions.py` 中的 validator 做。第二层只是尽量先自我修正，减少无效响应。

### `decision.py`

定义策略建议和决策记录的数据结构。

`StrategyAdvice` 是策略层给 prompt 的建议。

`DecisionRecord` 是每次 agent 行动后的完整决策记录，包含：

- 天数、阶段、玩家、角色。
- 动作类型。
- 候选目标。
- 最终 target 或 choice。
- 公开文本。
- 私有推理。
- 备选目标。
- 排除理由。
- 当时的 belief 快照。
- 当时的 memory 摘要。
- 决策来源：
  - `llm`
  - `fallback`
  - `policy_adjusted`

这份数据会写入 `logs/gameX.agent.jsonl`，供 UI 展开“决策过程”。

### `decision_log.py`

agent 决策日志收集器。

核心类：

```python
AgentDecisionRecorder
```

它保存所有玩家的 `DecisionRecord`，并可以导出为 JSONL：

```python
write_jsonl("logs/game14.agent.jsonl")
```

`collect_agent_decisions(agents)` 可以从一组 agent 中收集它们 runtime 里的历史决策。

当前运行 LLM 对局时，通常是创建一个共享 `AgentDecisionRecorder`，传给所有 `LLMPlayerAgent`。每个玩家行动后，自己的决策会追加进去。

导出第二层常用对象：

- `AgentMemory`

## `strategies/` 详细结构

```text
strategies/
  base.py
  villager.py
  werewolf.py
  white_wolf_king.py
  seer.py
  witch.py
  guard.py
  hunter.py
  __init__.py
```

`base.py` 提供通用策略：

- 投票、PK、猎人开枪：优先考虑高怀疑对象。
- 白天发言、PK 发言、遗言：表达站边、怀疑对象和投票倾向。
- 警长竞选：判断自己是否适合争警徽。
- 退水：避免无意义集体退水。

各角色策略在此基础上追加身份特有目标。

这层的好处是：prompt 不是完全从零让模型自由发挥，而是先给它一个“角色目标”和“候选目标倾向”。这样 agent 会更像真实狼人杀玩家，而不是每次随机决策。

## 每个玩家是否有独立记忆

是的。

每个 `LLMPlayerAgent` 都有自己的：

```text
LLMPlayerAgent
  -> AgentRuntime
      -> AgentMemory
      -> BeliefState
      -> RoleStrategy
```

所以 12 个玩家是 12 份记忆、12 份主观判断。

它们不是共享一个 memory。

但它们看到的公开日志来源相同，都是第一层的 `Observation.public_log`。区别在于：

- 每个玩家接收 request 的时机不同。
- 每个玩家的身份不同。
- 每个玩家有不同私有信息。
- 每个玩家过去自己的行动不同。
- 每个玩家 belief 更新出来的结果也可能不同。

这就形成了不同玩家的独立视角。

## 私有信息如何隔离

私有信息隔离主要依赖第一层的 `Observation`。

第二层不会直接读取完整 `GameState`。

它只能看到 request 里给它的：

```python
request.observation
```

例如：

- 狼人能看到 `known_roles` 中的狼队友。
- 预言家能看到自己的 `seer_checks`。
- 普通村民看不到狼队友和查验结果。

第二层 prompt 里明确要求：

```text
你只能根据自己可见的信息行动，不能假设上帝视角。
不要在公开发言中泄露你不可公开解释的私有视角。
```

同时，第二层会把 `private_reasoning` 和 `public_text` 分开：

- `private_reasoning`：给日志、上帝视角、调试看。
- `public_text` / `text`：规则层公开给其他玩家听。

## 决策过程如何记录

模型必须输出：

```json
{
  "text": "公开内容",
  "reasoning": "私有推理",
  "alternatives": [2, 4, 7],
  "rejected_reasons": ["为什么不选别人"]
}
```

`parsing.py` 会解析这些字段。

`runtime.py` 会补充：

- 当时的天数。
- 当前阶段。
- 玩家编号。
- 角色。
- 候选目标。
- 最终选择。
- belief 快照。
- memory 摘要。
- 来源类型。

最后形成 `DecisionRecord`。

`LLMPlayerAgent.act()` 会把新增的 `DecisionRecord` 写入共享 `AgentDecisionRecorder`。

对局结束时，`demo_llm.py` 或 UI 后端会写出：

```text
logs/gameX.agent.jsonl
```

UI 中每一步旁边的小三角展开看到的就是这里的内容。

## 第二层如何参与一局游戏

以命令行 LLM demo 为例：

```python
roles = random_standard_roles(seed=seed)
decision_recorder = AgentDecisionRecorder()
agents = create_llm_demo_agents(roles, decision_recorder=decision_recorder)
engine = GameEngine(roles, agents)
winner = await engine.run_until_finished(max_days=20)
```

这里发生了：

1. 第一层随机身份。
2. 第二层根据身份创建 12 个 agent。
3. 第一层 `GameEngine` 持有这些 agent。
4. 游戏推进时，第一层不断向对应玩家发 `ActionRequest`。
5. 第二层返回 `ActionResponse`。
6. 第一层结算。
7. 第二层记录每次决策过程。

## 当前第二层的优点

1. 和第一层解耦。
   第二层只依赖 `ActionRequest` / `ActionResponse`，不直接控制规则。

2. 每个玩家独立运行时。
   每个玩家有自己的记忆、信念、性格和策略。

3. 有基础局势判断。
   不是纯随机动作，会根据公开日志、私有信息、票型、身份声明更新 belief。

4. 有角色策略。
   不同身份的目标不同，例如狼人隐藏视角，预言家围绕查验，女巫谨慎用药。

5. 有失败兜底。
   模型失败或输出非法时，第二层会返回一个可用动作，不让游戏中断。

6. 有决策日志。
   每一步动作都能记录私有推理、备选项和排除理由，方便 UI 展示和后续优化。

7. 模型适配相对独立。
   只要实现 `ModelAdapter.complete()`，就可以换模型来源。

## 当前仍可继续优化的方向

1. 引入更强的 persona/profile 系统。
   现在只有简单座位轮换性格，后续可以设计更稳定的玩家风格，例如谨慎型、强势型、逻辑型、摇摆型，并影响上警、发言、投票和冒险程度。

2. 强化长期反思。
   当前有 `decision_history`，但还没有赛后反思、自我进化、跨局学习。后续可以增加 reflection 模块。

3. 更精确的 belief 更新。
   当前怀疑度更新是简单启发式。后续可以加入票型链、发言一致性、身份对跳、死亡收益分析。

4. 减少重复发言。
   LLM 有时会模仿或复读前文。可以在 prompt 或后处理里加入“避免重复他人原话”的约束。

5. 更强的合法性自检。
   现在有 policy 和第一层 validator，但 agent 内部可以再加一层“输出前自检”。

6. 引入 LangGraph。
   如果想让玩家思考流程更清晰，可以把 `AgentRuntime.act()` 拆成 LangGraph 节点：
   `observe -> memory -> belief -> strategy -> generate -> validate -> reflect -> output`。

7. 支持人类玩家混合。
   第二层可以新增 `HumanPlayerAgent`，它同样接收 `ActionRequest`，但通过 UI 等待真人输入，再返回 `ActionResponse`。

## 一句话总结

第二层 `playeragent/` 是每个玩家的大脑。它把第一层发来的规则动作请求，转化为“带记忆、带主观判断、带角色策略、带私有推理记录”的玩家决策。它不改变规则，只决定玩家在规则允许范围内如何行动。
